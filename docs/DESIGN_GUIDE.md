# Design Guide: LTE NMS

Этот документ задает единые правила проектирования и разработки для текущего проекта.

Он нужен, чтобы:
- все участники писали код в одном стиле;
- архитектура не расползалась при добавлении функций;
- переход от MVP к следующему этапу был без полной переработки.

## 1. Цель проекта

Минимальная версия системы управления LTE сетью должна уметь:
- подключать узлы по `NETCONF`;
- хранить состояние узлов;
- читать базовый инвентарь;
- выполнять безопасное изменение конфигурации;
- принимать события и heartbeat;
- отдавать данные через HTTP API.

## 2. Основные принципы проектирования

1. **Просто и понятно:** сначала рабочий минимум, потом расширение.
2. **Одна зона ответственности на сервис:** `inventory` не должен менять конфигурацию, `cm` не должен хранить PM логику.
3. **Явные контракты:** все входы/выходы оформлены структурами и проверками.
4. **Без скрытой магии:** если есть правило, оно должно быть видно в коде и документации.
5. **Расширяемость без слома:** новые функции добавляются новыми методами/пакетами, а не переписыванием базовых слоев.

## 3. Слои проекта

Текущая структура:

- `cmd/nms-lte` — запуск приложения.
- `internal/app` — сборка зависимостей и сервер.
- `internal/httpapi` — HTTP методы и маршруты.
- `internal/service/*` — бизнес-логика по доменам.
- `internal/store/memory` — хранилище данных (в MVP in-memory).
- `internal/model` — общие модели данных.

Правило зависимостей:

- `httpapi` знает о `service`.
- `service` знает о `store` и `model`.
- `store` знает только о `model`.
- `model` не зависит ни от кого.

## 4. Правила по сервисам

### 4.1 `ne` (сервис узлов)
- Отвечает за регистрацию узлов и их состояние.
- Не должен выполнять CM и PM операции.

### 4.2 `inventory`
- Отвечает за чтение и сохранение снимка инвентаря.
- Работает только с данными модели, без изменения конфигурации.

### 4.3 `cm`
- Отвечает только за операции изменения конфигурации.
- Обязательные шаги: `lock -> edit-config -> validate -> commit -> unlock`.
- Любая операция должна иметь журнал шагов.

### 4.4 `fault`
- Отвечает за события, их хранение и heartbeat.
- Если heartbeat не прошел, событие должно быть создано автоматически.

### 4.5 `pm`
- Отвечает за сбор метрик и чтение метрик.
- В MVP допускается простой формат метрик без сложной агрегации.

## 5. Правила API

### 5.1 Общие
- API работает по `JSON`.
- Все ошибки возвращаются в едином формате: `{ "error": "..." }`.
- Для неизвестного маршрута — `404`.
- Для неверного метода — `405`.

### 5.2 Коды ответа
- `200` — успешное чтение/операция.
- `201` — создана новая сущность.
- `400` — ошибка входных данных.
- `404` — сущность не найдена.
- `409` — конфликт выполнения операции (например, CM в статусе failed).

### 5.3 Имена маршрутов
- Используем единый префикс: `/api/v1/...`.
- Путь должен быть коротким и предметным: `ne`, `cm/requests`, `fault/events`, `pm/samples`.

## 6. Правила моделей и данных

1. Каждая модель содержит понятный идентификатор `ID`.
2. Для событий и операций фиксируем время (`CreatedAt`, `UpdatedAt`).
3. Статусы используем из ограниченного набора (`active`, `failed`, `success`, `running`).
4. В `DN` и связанных идентификаторах соблюдаем единый формат именования.

## 7. Правила обработки ошибок

- Ошибка должна быть:
  - ранней (обнаружить как можно раньше);
  - понятной (короткий текст без лишнего);
  - полезной для API пользователя.
- Нельзя скрывать ошибку, если операция не завершена.
- Нельзя возвращать `200`, если логически есть ошибка.

## 8. Логирование и наблюдаемость

Минимум для каждого важного действия:
- имя операции;
- идентификатор узла (`ne_id`), если есть;
- результат (`success`/`failed`);
- время выполнения.

При расширении проекта добавляем:
- метрики Prometheus;
- трассировку по `trace_id`;
- дашборд по `CM`, `fault`, `PM`, `inventory`.

## 9. Тестовая стратегия

### 9.1 Что тестируем обязательно
- успешный и ошибочный сценарий CM;
- синхронизацию инвентаря;
- базовые ошибки API (невалидный JSON, неизвестный `ne_id`).

### 9.2 Уровни тестов
- Unit-тесты для сервисов.
- API-тесты для ключевых маршрутов.

### 9.3 Критерий готовности фичи
- код отформатирован (`gofmt`);
- тесты проходят (`go test ./...`);
- есть обновление документации, если добавлен новый API или статус.

## 10. Правила для студентов

### Студент 1 (сеть и NETCONF)
- отвечает за все, что связано с подключением к узлам;
- не меняет бизнес-логику API без согласования;
- документирует ограничения и особенности подключений.

### Студент 2 (API и данные)
- отвечает за HTTP API, модели и хранение;
- не меняет сетевой слой без согласования;
- следит за единым форматом ошибок и ответов.

### Совместные правила
- перед изменением контракта API обновлять документацию;
- крупные архитектурные изменения фиксировать в коротком ADR;
- избегать «быстрых» решений, которые ломают структуру слоев.

## 11. Как расширять MVP дальше

Порядок развития:
1. Перенести `store` из памяти в PostgreSQL.
2. Добавить реальный NETCONF клиент вместо учебной заглушки.
3. Ввести пакетные операции CM.
4. Расширить PM и fault-корреляцию.
5. Добавить bulk XML сценарии с учетом `TS 28.659`.

Главное правило: расширять через новые компоненты, а не переписывать базовые слои.
